# 🎱 Billiard Club — Telegram Web App

Клубный рейтинг по Эло с P2P-подтверждением матчей, заявками на игру и панелью администратора.
4 языка (EN / PL / UK / RU), рассчитано на **300+ одновременных пользователей**.

## Структура

```
server/   Node.js + Fastify + SQLite (WAL) — API, авторизация, бизнес-логика
webapp/   React + Vite — Telegram Mini App (тёмная iOS-стилистика)
```

## Архитектура

```
Telegram-клиент
   │  initData (подписан HMAC ботом)
   ▼
webapp (React SPA) ──HTTP──► Fastify API ──► SQLite (WAL)
   ▲                            │
   └── поллинг /matches/active ◄┘  (P2P-подтверждение, 1.5–15 c адаптивно)
```

### Почему это держит 300+ онлайн

- **SQLite в WAL-режиме + один процесс Node**: все транзакции синхронные и
  однопоточные — гонки между игроками исключены архитектурно, а пропускная
  способность (тысячи запросов/сек на чтение) на порядок выше нужной:
  300 клиентов × поллинг раз в 4 с ≈ 75 rps.
- **Адаптивный поллинг**: 1.5 с только во время активного матча, 4 с при
  чекине, 15 с в остальное время.
- **Кэши**: рейтинг (5 с), результаты HMAC-проверки initData (5 мин LRU).
- **Препарированные SQL-выражения** с кэшем и индексы на всех горячих путях.
- Путь масштабирования дальше (1000+): вынести БД в PostgreSQL и поллинг
  в WebSocket/SSE — слой `db.js` и маршруты это позволяют.

### Модель данных (ключевое)

Статусы **не хранятся как boolean** — хранятся timestamp'ы
`activated_until` / `checked_in_until`, а `is_activated` / `is_checked_in`
вычисляются от текущего времени при каждом чтении. «Протухшие» флаги невозможны.

| Таблица    | Назначение |
|------------|-----------|
| `users`    | профиль, Эло, роль, сроки активации/чекина, состояние поиска |
| `matches`  | конечный автомат: `pending → confirmed / conflict / timeout / cancelled` |
| `requests` | заявки на 3 дня (уникальность: игрок+день+слот) |
| `favorites`| избранное |
| `audit_log`| все действия админов |

### P2P-фиксация матча (30 секунд)

1. Игрок А (с активным чекином) жмёт «Победа/Поражение» в профиле Б →
   `POST /matches/report` → строка `pending` с `expires_at = now + 30 c`.
2. Игрок Б видит входящее подтверждение (поллинг) и отвечает →
   `POST /matches/:id/respond` — в **одной транзакции** проверяется статус,
   срок, зеркальность, пересчитывается Эло (K=32, zero-sum) и обновляются оба профиля.
3. Исходы: зеркально → `confirmed` (модалка «было → стало ±N»); совпали →
   `conflict`; не успел → `timeout` (ленивый sweep + фоновый каждые 15 с);
   инициатор отменил → `cancelled`. В трёх последних Эло не меняется.
4. Непросмотренные итоги доставляются через `unseen` + `ack` — игрок увидит
   результат, даже если закрывал приложение.

### Безопасность

- **Подпись Telegram initData** проверяется на каждом запросе
  (HMAC-SHA256, `timingSafeEqual`, свежесть `auth_date` ≤ 24 ч). Подделать
  личность без токена бота невозможно. Покрыто тестом `server/test-auth.mjs`.
- **Все права — только на сервере**: роль admin из БД, чекин/активация по
  timestamp'ам; клиентские блокировки кнопок — лишь UX.
- **Валидация** всех тел запросов JSON-схемами Fastify + очистка строк
  (управляющие символы, длина), `bodyLimit` 16 КБ.
- **Rate limit**: 240/мин на пользователя глобально, жёстче на onboard (5/мин)
  и report (12/мин).
- **Контакты игроков** видны только активированным участникам клуба.
- **Деактивация** атомарно отзывает всё: активацию, чекин, поиск, заявки,
  незавершённые матчи. Действия админов пишутся в `audit_log` (хранится 180 дней).
- SQL только через параметризованные prepared statements; React экранирует
  вывод (XSS); security-заголовки; CORS в проде выключен (фронт отдаёт сам сервер).

## Запуск

### Разработка (без Telegram)

```bash
# терминал 1 — API (личность подменяется заголовком x-dev-tg-id)
cd server
copy .env.example .env        # DEV_AUTH=1, BOT_TOKEN можно не задавать
npm install
npm run dev

# терминал 2 — фронтенд (vite проксирует /api на :3000)
cd webapp
npm install
npm run dev                   # http://localhost:5173
```

В браузере личность = `localStorage.devTgId` (по умолчанию `1`).
Чтобы стать админом: добавьте свой dev-id в `ADMIN_TG_IDS` в `.env`.
Сбросить данные — удалить `server/data/`.

### Продакшен

```bash
cd webapp && npm install && npm run build   # соберёт webapp/dist
cd ../server && npm install
# .env: BOT_TOKEN=<из @BotFather>, ADMIN_TG_IDS=<tg id админов>, DEV_AUTH=0
npm start                                   # отдаёт и API, и собранный фронт
```

1. Поставьте перед сервером HTTPS (Caddy/nginx + certbot) — Telegram требует TLS.
2. В @BotFather: `/newapp` (или кнопка меню бота) → URL вашего домена.
3. Админы заходят в приложение → получают роль автоматически по `ADMIN_TG_IDS`
   → чекинят игроков в «Панели администратора».

## Бизнес-правила (сводка)

| Механика | Срок | Что даёт |
|---|---|---|
| Активация (админ) | 45 дней | поиск, заявки, лента, контакты |
| Чекин (админ) | 6 часов | кнопки «Победа/Поражение»; продлевает активацию на 45 дней |
| Подтверждение матча | 30 секунд | зеркальный ответ соперника |
| Заявки | 3 дня × 2 слота | видны активированным с контактом |
| Поиск «на сегодня» | до полуночи | список ищущих с контактами |
| Эло | K=32, старт 1000 | пересчёт только при подтверждённом матче |
